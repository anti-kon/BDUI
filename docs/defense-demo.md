# Сценарий демонстрации BDUI для защиты

Документ описывает короткую демонстрацию, которую можно провести за 5-7 минут
после успешного запуска локальных проверок.

## Перед демонстрацией

1. Убедиться, что используется Node.js 24 и npm 11:

```bash
node -v
npm -v
```

2. Установить зависимости и собрать проект:

```bash
npm ci
npm --prefix examples/task-manager ci
npm run verify:all
```

3. Сгенерировать актуальные скриншоты для диплома:

```bash
npm run capture:defense-screenshots
```

Скриншоты сохраняются в `docs/assets/`:

- `taskly-dashboard.png` - основной экран Taskly;
- `taskly-settings-dark.png` - настройки Taskly в темной теме;
- `web-demo-settings.png` - web-demo с рендерингом Campus-контракта.

## Демонстрация 1. Полный продуктовый пример Taskly

1. Собрать и запустить standalone-приложение:

```bash
npm run verify:examples
npm --prefix examples/task-manager start
```

2. Открыть `http://127.0.0.1:4000`.

3. Показать главный экран:

- контракт загружен и кэшируется;
- UI полностью приходит из JSON-контракта;
- кнопки запускают декларативные SAL-действия.

4. Нажать `Обновить данные` и `Загрузить каталог`.

Что проговорить: рендерер не содержит бизнес-экраны в коде, он исполняет
контракт, источники данных и действия.

5. Открыть `Новая заявка` и пройти flow из трех шагов:

- выбрать тип и срочность;
- заполнить описание и бюджет;
- отправить заявку.

Что проговорить: многошаговый сценарий описан в контракте, а переходы,
валидация и итоговое состояние исполняются рантаймом.

6. Открыть `Настройки`:

- переключить тему `Светлая` / `Темная`;
- изменить поле `Имя`;
- показать, что select меняет значение, а фокус в поле не сбрасывается.

## Демонстрация 2. Web-demo как независимый рендер контракта

1. Запустить статическую демонстрацию:

```bash
npm run demo:web
```

2. Открыть URL, который выведет `serve`, обычно `http://localhost:3000`.

3. Показать маршруты `Главная`, `Расписание`, `Задания`, `Настройки`.

Что проговорить: браузер получает готовый JSON-контракт и import map с
vendored-пакетами, а значит демонстрация проверяет реальную ESM-сборку, а не
только unit-тесты.

## Доказательства завершенности

Перед защитой стоит открыть GitHub Actions и показать зеленые jobs:

- `verify` - сборка, генерация, typecheck, lint, prettier, тесты, coverage,
  контракты, web-demo, browser smoke, Taskly smoke, production audit;
- `android native build` - сборка Android Compose prototype;
- `ios native typecheck` - typecheck SwiftUI renderer.

Если нет Android/iOS-устройства на защите, это честно формулируется как
ограничение среды: код и CI-gates подготовлены, а физический smoke на устройстве
относится к аппаратному acceptance.
